# Current Progress

This page is the release-facing status summary for the current `main` branch.
It avoids historical migration notes and describes only the active ChemWorld
architecture.

## Stage Summary

ChemWorld-Bench is now an alpha research benchmark with a unified Gymnasium
environment:

```python
import gymnasium as gym
import chemworld

env = gym.make("ChemWorld", task_id="reaction-optimization-standard", seed=42)
obs, info = env.reset()
obs, reward, terminated, truncated, info = env.step(
    {"operation": "add_solvent", "volume_L": 0.03, "solvent": 2}
)
```

The project is no longer organized around a single batch-reactor script. The
active runtime uses task profiles, operation kernels, domain services,
mechanism manifests, typed ledgers, transaction records, observation services,
scoring services, and replay verification.

## Implemented Capabilities

The current codebase provides:

- one formal Gymnasium entry point: `ChemWorld`;
- one shared world law: `chemworld-physical-chemistry`;
- 14 registered task slices over that world law;
- task cards generated from the task registry;
- scenario cards and mechanism hashes for reproducibility;
- action validation, operation masks, safety/cost signals, and wrappers;
- typed state ledgers for material, phase, vessel, equipment, thermal, and
  process state;
- event-driven operation execution with transaction records and constitution
  checks;
- reaction, phase partition, separation, crystallization, distillation,
  continuous-flow, electrochemistry, instrument, and scoring services;
- partial observations with `observed_mask`, `raw_signal`,
  `processed_estimate`, and `uncertainty`;
- virtual HPLC, GC, UV-vis, IR, NMR, and final-assay signal packets;
- JSONL trajectory logging, replay verification, submission bundles,
  dataset export, local evaluation-machine scaffolding, and baseline reports;
- official baseline families including random, LHS, greedy, GP BO, RF EI,
  safe GP BO, scripted chemistry, LLM replay, and LLM planner adapters;
- documentation, example scripts, and a 12-day tutorial notebook sequence.

## Registered Task Families

The current registry contains reaction-only, reaction-to-assay,
reaction-to-purification, partition, crystallization, distillation,
continuous-flow, electrochemistry, tool-agent, safety, explanation, and
generalization tasks. Task details are generated by:

```bash
chemworld tasks list
chemworld tasks show reaction-to-purification
chemworld tasks card reaction-to-purification
```

Use [Task Cards](task_cards.md) for the compact registry matrix and
[Benchmark Protocol](benchmark_protocol.md) for scoring and submission rules.

## Maturity Boundary

ChemWorld is a virtual benchmark, not a real reaction predictor or production
process simulator. Current modules have explicit maturity levels:

- `lite`: benchmark ODE or simplified physical model with tests;
- `reference_validated`: a local implementation checked against analytical
  or reference-style cases;
- `proxy`: a qualitative process model kept for exploratory interaction tasks;
- `professional_candidate`: a deeper model with evidence but not yet a full
  professional simulator replacement.

Every task, trajectory, baseline report, and submission summary must carry
`kernel_maturity`, `physics_maturity`, and `proxy_allowed` metadata.

## Known Risks

- Some exploratory tasks intentionally allow proxy process modules.
- The broader reaction-to-purification/tool-agent task slices currently expose
  many process operations; future task-profile hardening should narrow these
  slices when the benchmark contract is frozen.
- Private evaluation is still implemented as a local maintainer-salt workflow,
  not as a hosted hidden server.
- Baseline tables are available through local report generation, but the
  public paper-release table still needs to be frozen and versioned.

## Recommended Next Work

Near-term benchmark hardening:

- freeze task cards and public seeds for the pre-release task set;
- run the baseline report generator for all release tasks;
- narrow over-broad task profiles where a task should not expose unrelated
  process operations;
- add signed private-eval artifacts for any public leaderboard claim;
- keep the environment self-consistency audit in the release gate.

Near-term research work:

- compare human, BO, LLM replay, and human-plus-LLM strategies;
- study public-test to private-eval generalization;
- analyze safety-aware exploration behavior;
- analyze whether mechanism explanations correlate with optimization quality.
