# 站点层级规划

当前站点采用“导航先稳定、文件路径后迁移”的策略。也就是说，先在
`mkdocs.yml` 中固定读者看到的信息架构，暂时不批量移动 `docs/` 下的文件路径。
这样可以避免大量断链，同时让公开站点先服务科研预发布。

## 组织原则

站点不按开发历史组织，而按外部使用路径组织：

1. 先说明当前预发布版本能做什么、不能做什么。
2. 再给 benchmark contract：任务、种子、baseline、提交、评测和 release gate。
3. 然后给 agent 接入、数据产物和世界底座。
4. 最后保留审计、路线图和开发协作材料。

这个顺序的目标是让外部研究者先看到稳定 contract，而不是先进入内部实现细节。

## 站点交互约定

- 顶部右侧使用 Material 原生语言切换入口；首页正文不再放中英文切换按钮。
- 主导航保持在左侧抽屉/侧栏；页面内标题目录保持 Material 默认右侧 outline。
- 右侧 outline 可以折叠，但不替代左侧主导航。
- URL 暂时保持当前扁平文件路径，避免影响已发布链接。

## 后续迁移规则

如果后续要把 `docs/` 改成分目录结构，按下面顺序执行：

1. 先迁移低耦合页面：`current_progress.md`、`demos.md`、`roadmap.md`、`project_management.md`。
2. 再迁移 benchmark contract：`tasks.md`、`task_cards.md`、`benchmark_protocol.md`、`submission.md`。
3. 最后迁移 foundation 和 audit 页面。
4. 每批迁移后运行 `python -m mkdocs build --strict`，并检查公开链接。
5. 如果站点已被外部引用，再引入 redirect，而不是直接打断旧 URL。
